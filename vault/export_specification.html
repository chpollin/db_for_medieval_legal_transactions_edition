CSV Format¶

  • Delimiter: Semicolon (;)
  • Line endings: CRLF (\r\n)
  • Encoding: UTF-8 (no BOM)
  • Quoting: None — semicolons in free text produce broken rows
  • Header: First line is column names
  • Trailing newline: Only filenames.csv has a trailing CRLF; all others do not

Data Quality Issues¶

1. Trailing Semicolon in persons.csv¶

Every data row has 17 semicolon-delimited fields (16 columns + 1 trailing empty field). The header has only 16 fields. LOAD DATA must use @dummy to absorb the 17th empty field.

Present in both curated and uncurated output. Produced by the pipeline itself.

2. addname_orig / addname_reg Column Swap (persons.csv)¶

The column headers addname_orig and addname_reg are swapped relative to actual content: addname_orig contains regularized values, addname_reg contains original values. This is a preserved XSLT bug. The existing frontend consumes them as-is.

3. Defective CSV Rows (Semicolons in Free Text)¶

The CSV writer uses no quoting. Semicolons in free text split fields incorrectly.

4. kin_relations UTF-7 Encoding (data_export only)¶

The file data_export/.../kin_relations_in_sources.csv is encoded in UTF-7 (semicolons appear as +ADs-, underscores as +AF8-). This is a legacy XSLT output artifact. The Python pipeline output (pipeline/output/kin_relations_in_sources.csv) is normal UTF-8.

Recommendation: Always use the pipeline output (pipeline/output/) or a re-exported curated version. The data_export UTF-7 file cannot be imported with standard LOAD DATA.

5. Empty Tables¶

Three tables have 0 data rows in the curated export: - buis_relations_in_sources — 0 rows in both curated and uncurated - tenant_relations_in_sources — 0 rows in both curated and uncurated - title_ref_relations_in_sources — 0 curated, 7 uncurated (filtered by QGW-1414 curation)

These represent annotation types that were defined but not (yet) used in the sources. The table structures are retained for future use.

6. Always-Empty Columns¶

The following columns have max_length = 0 across all rows: - events_in_sources.date_orig - places.parzelle_idno - kin_relations_in_sources.select_key - friend_relations_in_sources.cert_to_related_key, select_key - rep_relations_in_sources.cert_to_related_key, select_key - topo_relations_in_sources.cert_place_key, cert_topo_relation_key - owner_relations_in_sources.cert_place_key - places_in_events.event_catchword

All retained for structural consistency with the pipeline.

7. Pipe-Separated Multi-Value Columns¶

Several columns contain pipe-separated (" | " or "|") values for multi-valued fields. These are denormalized by design (the existing PHP frontend parses them as strings). Documented in the "Pipe-separated" column of each table specification above.

8. Orphaned Foreign Keys¶

  • 4,085 orphaned ev__ references — eventList.xml was deleted; all event_key values are referentially orphaned. The event_key column is still meaningful as a grouping key, just not resolvable against a master table.
  • 9,215 never-referenced register IDs — pe__: 6,342; pl__: 2,493; org__: 380. Master data entries that are not referenced in any source annotation.

FK constraints must remain commented out to allow import.

Curation Filter (historical, discontinued)¶

K6 resolved: no curation filter. All collections processed unconditionally. The legacy filter removed 819 files (QGW > 1414) from data_export/.

Future: SQL Import Notes¶

Should a SQL import be needed, the following index strategy is recommended: - PK: id on all tables - FK indexes: file_key/xml_key, person_key, org_key, place_key, event_key on all referencing tables - Query indexes: date_not_before/date_not_after (date range), event_role (role filter), status (done/in progress), kind_of_linking (ref/corresp) - FK constraints: Must remain commented out (4,085 orphaned ev__ refs, 9,215 unreferenced register IDs)
