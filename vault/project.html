Project and Research Context¶

Description of the project, data model, and annotation methodology. Primary source for Gutachten conditions #2 (ODD), #5 (individualisation), and #6 (encoding guidelines).

1. Project Context¶

The database db_for_medieval_legal_transactions is a TEI-XML-based prosopographic database for medieval Viennese legal transactions (ca. 1177--1524). The data model follows the factoid approach of digital prosopography.

Origin: - Master thesis (real estate transfers Vienna 1360--1373, Excel) - Conceptualisation of XML-based database - Dissertation "Im Netz der Stadt" (Gruenwald) -- full DTD created here - Parallel use in the project "Stadt und Gemeinschaft" (Lutter, University of Vienna) - Light DTD was designed but never deployed in DOCTYPE declarations

Reference models: Factoid prosopography (KCL, factoid-dighum.kcl.ac.uk), TEI P5, CIDOC-CRM (long-term).

Publication: DHd blog post "Die digitale Erfassung von mittelalterlichen Rechtsgeschaeften" (Gruenwald, 31 Oct 2021, dhd-blog.org/?p=16737).

RelaxNG Schema¶

RelaxNG schema sources/schema/toolbox.rng (~860 lines) replaces the DTD as active schema (Gutachten requirement #2). Hand-written from empirical analysis because: (1) trang converter requires Java (unavailable), (2) DTD was bloated with ODD artifacts (133 defined elements, only 48 used).

Architecture — 12 sections:

Section Content
1 6 attribute groups (att.global, att.typed, att.datable.w3c, att.declarable, att.docStatus, att.sortable)
2 2 content macros (macro.paraContent = 18 inline elements, macro.phraseSeq = 16)
3--6 Root (TEI), Header, Manuscript description, Facsimile
7--8 Body/Text, Annotation model (rs, roleName, triggerstring, catchwords)
9--10 Inline elements (p, add, reg, etc.), Bibliographic (bibl, title, etc.)
11 XInclude (own namespace)
12 Documentation of 85 excluded elements

Attribute conflict resolutions (RelaxNG forbids duplicate attribute definitions):

Attribute Conflict Resolution
cert att.global vs. rs/roleName enum Removed from att.global, per-element on rs/roleName
n att.global (text) vs. triggerstring/catchwords (enum) att.global NOT applied to triggerstring/catchwords
source att.global vs. reg/placeName Removed from att.global, per-element
resp att.global vs. div Removed from att.global, per-element on div/TEI

Constraint strategy (why some attributes use <text/> instead of enums):

Attribute Constraint Reason
rs/@type <text/> Data quality: off, rs, pe, witness, empty in ~20 files
rs/@ref <text/> 158 files with ref="NULL" placeholder
rs/@role Enum (7 values) Stable, no deviations in corpus
roleName/@type Enum (13 values, optional) "org" as 13th value found empirically; optional because 1 file lacks @type
triggerstring/@n Enum (4 values, optional) "impl_rules" as 4th value; optional because 7 files lack @n
date/@when <text/> Contains values like "99999999"

Content model deviations (DTD definition vs. empirical usage):

Element DTD Empirical Schema solution

3. Four-Level Annotation Model¶

Hierarchical annotation structure (each level is part of the higher one):

Level Subject TEI Element Categorisation
1. Events Legal transactions <rs type="event" ref="#ev__*"> Via <catchwords n="disp"> -- dispositive verbs extracted, categorised post-hoc
2. Functions Roles in legal transaction <rs type="fn" role="issuer\|recipient\|witness\|other"> For "other": <catchwords n="fn"> for specification
3. Entities Persons, places, organisations <rs type="person\|org\|place" ref="#*__*"> ID linkage to registers (XPointer shorthand)
4. Attributes/Relations Properties + connections <roleName type="*" corresp="#*__*"> Attributes: prof, title, dead; Relations: kin, buis, rep, owner, off, title_ref, staff, topo, friend

Key design decision: Event categorisation is not done a priori but through extraction of dispositive verbs and post-hoc category formation. This allows flexibility across heterogeneous source collections.

Place references: Places are referenced via two mechanisms: <rs type="place"> (49 occurrences) and @corresp containing pl__ in <roleName> (topo, owner relations). Both mechanisms combined reach only 44 of 2,537 place register entries -- places are systematically underrepresented in annotations (K5 resolved: annotation status quo).

4. ID Construction Schemes¶

Events¶

ev__ + source_abbreviation + (_) number
e.g.: ev__QGW_II_II_2870, ev__SB_CD_00642
Mentioned events: ev__QGW_II_II_00606_men_1

Persons¶

pe__ + normalised_forename + (_) normalised_surname + (_) source_abbreviation + (_) number
e.g.: pe__johann_maurperger_QGW_II_II_2870
Homonyms: pe__agnes_QGW_II_II_3250_a, pe__agnes_QGW_II_II_3250_b

Rules: No umlauts in IDs. Forenames via external normalisation table. Surnames via source register; if absent: original spelling. No surname → occupation. Women/children: potential surname added in <add>.

Organisations¶

org__ + settlement + (-) patron_saint/proper_name + (_) institution_type + (-) sub_org
e.g.: org__wien, org__wien-st_stephan, org__wien-st_stephan-altar_st_dorothea

Principle: From general to specific. Hierarchical nesting in orgList. Types: Stadt, Dioezese, Kirche/Kapelle, Pfarre, Spital, Kloster (m/f), Altar, Messe, Zeche/Bruderschaft, Herzogtum, Koenigreich, Reich, OTHER.

Places¶

pl__ + settlement + (-) name
e.g.: pl__wien, pl__wien-hoher_markt
Real estate: pl__ + settlement + (-) immo_ + street + (_) number
e.g.: pl__wien-immo_karntnerstrasse_1

Types: settlement, street, immo. Hierarchisation possible.

5. Seven-Step Tagging Workflow¶

  1. Insert editor initials (@resp)
  2. Mark up events (<rs type="event">)
  3. Mark dispositive verbs (<catchwords n="disp">) -- optionally with <add> for implicit information
  4. Assign functions (<rs type="fn" role="issuer|recipient|witness|other">) -- for "other": additionally <catchwords n="fn">
  5. Mark up entities -- first <rs/>, then assign IDs (@ref)
  6. Add attributes/relations (<roleName type="*" corresp="*__*">) -- incl. @select for spatial precision
  7. Mark up units of measurement (<measure/>)

Additional: @cert (high--low) for probability assessment, <unclear>, <add>.

This workflow is the de facto annotation guideline of the project and the substantive basis for Gutachten condition #6.

6. Registers¶

Register Entries Size ID Prefix Path
personList.xml 16,088 8.4 MB pe__ indices/
orgList.xml 1,078 381 KB org__ indices/
placeList.xml 2,537 1.4 MB pl__ indices/

Coverage: 9,215 of 19,703 register entries are never referenced from any source file. Breakdown: 6,342 persons, 2,493 places, 380 organisations (see pipeline Integrity Checks).

Register structure: - personList: <persName> (forename/reg, surname/reg, addName/reg, genName), <death notAfter="">, <idno> for external authorities. @sex (f/m), @source, @resp, @cert, @sameAs. - orgList: Hierarchical <org> elements, with <location>/<address>, <trait>/<label>/<desc>. Types via @type. - placeList: Places with GeoNames URIs (<idno>) and lat/lng coordinates (<geo>). Hierarchical places (houses within streets). Types: immo, settlement, street, river.

Schema: lists.xsd. Defines listPerson (@sex required f/m), listOrg (19 type values), listPlace (4 type values with <geo> and <idno>), listEvent (deprecated, eventList deleted).

Phase A corrections: 4 entries in personList with faulty ID prefix (pe_ instead of pe__ or missing prefix) corrected. 1 duplicate in orgList resolved (org__wien-st_stephan-altar_zwoelfapostel-messe_ybbs_anna, line 3726 type="Altar" vs. 3772 type="Messe").
