Digital Edition (Web Frontend)¶

Static HTML edition of the 3,041 TEI-XML source documents, built as browsable, annotated web pages for GitHub Pages hosting.

Architecture¶

Static site generator approach: Python/lxml + Jinja2 builds HTML from TEI-XML sources. No server required — output is pure HTML/CSS/JS served from docs/.

Build Commands¶

# Full build (all documents + index)
python -m edition build

# Single file (for development/testing)
python -m edition build --single sources/QGW/Vienna_1177-1414_ready/done/100.xml

# Run tests
python -m pytest edition/tests/ -v

Dependencies¶

  • lxml>=4.9 (already in pipeline)
  • jinja2>=3.1 (new dependency)
  • Reuses: pipeline.config, pipeline.utils.xml_loader, pipeline.utils.text_utils, pipeline.utils.date_parser

Build Architecture¶

Two-pass build to compute cross-document data (prev/next links + reverse index require all metadata first):

  1. Parse pass: _parse_file() — extract metadata via _xpath_text() helper + render body HTML + extract entity refs
  2. Compute: _compute_prev_next() — sort documents per collection, assign sequential links
  3. Reverse index: _extract_entity_refs(root) scans <rs type="person|org|place" ref="..."> + <roleName corresp="..."> per document. Aggregated into {entity_id → [doc_meta_list]} after pass 1. Sorted by date.
  4. Write pass: _write_file() — render Jinja2 templates with full context
  5. Index: _build_index() — generate timeline data (decade buckets + min_year/max_year), place frequencies, search JSON
  6. Register pages: _build_register_list_pages() — three list pages with embedded search JSON. _build_register_detail_pages() — one HTML page per entity (19,699 total) with reverse index data

Helper functions: _xpath_text(root, xpath, default) consolidates repeated XPath→text extraction. _load_registers() deduplicates register loading (with timing) between build_single() and build_all().

Performance¶

  • 3,041 documents + 19,699 register detail pages in ~70s
  • Register loading: ~1.8s (16,084 persons, 1,078 orgs, 2,537 places)
  • Reverse index: 10,339 entities linked to documents
  • 82 tests, all passing

TEI → HTML Rendering¶

The renderer (renderer.py) walks the TEI element tree recursively using a dispatch table (ELEMENT_HANDLERS). Entity rendering (person/org/place) uses a shared _ENTITY_MAP dispatch to avoid triplication. Templates only handle page layout; the renderer builds the HTML string for the annotated document body.

Tooltips¶

Entity spans get title attributes from register lookup: - Person: "Forename Surname (d. DeathDate) [pe__id]" - Organisation: "Name (Type) [org__id]" - Place: "Name (Type) [lat, lng] [pl__id]" - Orphaned refs: Show raw ID gracefully (4,085 orphaned event refs)

JavaScript enhances native tooltips with styled popups showing type badge and ID.

UI Design (Shneiderman's Information Seeking Mantra)¶

"Overview first, zoom and filter, then details-on-demand"

Index Page (overview + filter)¶

  • Corpus summary line: document/person/org counts + date range
  • Dual-handle range slider: histogram background (decade bars) + two overlaid <input type="range"> for min/max year filtering. Labels track handle positions dynamically
  • Collection chips: inline pills replacing large cards, click to toggle filter
  • Faceted filters: place dropdown (top 15), facsimile (with/without), text search (multi-word AND)
  • Active filter chips: removable pills showing current filters
  • Progressive table rendering: IntersectionObserver loads 100 rows at a time (no hard cap)
  • Row preview (details-on-demand): click row to expand inline preview with full regest + facsimile thumbnail

Document Page (Text-Bild-Synopse)¶

  • Synopsis layout: text + facsimile side by side as default view (CSS grid 1fr 1fr), no tab switching needed
  • Compact toolbar (sticky below nav): breadcrumb, Nr. + date + place, inline prev/next
  • Facsimile panel: always visible for documents with facsimile URLs (79%), sticky position, lazy-loaded images, page navigation, zoom/pan
  • Compact details section (below content): metadata (Archiv, Originaldatierung, Quelle, Digitalisat link) + annotation legend in two-column grid
  • No-facsimile fallback (21%): single column, text body centred at max-width: 52rem
  • Prev/next navigation: inline in toolbar + bottom of page
  • Print metadata: hidden <div class="doc-print-meta"> block, shown only in @media print

Register Pages¶

  • List pages (persons.html, organisations.html, places.html): alphabet bar, search, type/docs filter, sortable table, progressive rendering. JSON data embedded, same client-side pattern as index page.
  • Detail pages (~19,699 under docs/register/): entity metadata + reverse-index document table. One page per entity.

Design Philosophy: Edition as Epistemic Infrastructure¶

The edition is not a polished end-product that hides its construction. It is a transparent research tool that makes the entire data pipeline visible and traceable.

Principle: Two-Layer UI¶

Every data point in the UI has two layers:

  1. Reading layer (primary): Human-readable, scholarly labels for external researchers
  2. Data layer (secondary): Technical identifiers, file paths, pipeline provenance for traceability

Both layers are always present. The data layer is visually subordinate (smaller, monospace) but never hidden. There is no toggle — both layers are always visible by design.

Why This Matters¶

  • Reproducibility: Any annotation, entity reference, or structural element can be traced back to its TEI-XML source and the pipeline step that produced it
  • Quality control: The contractor (DHCraft) and the client can verify data processing by inspecting provenance directly in the edition
  • Methodological transparency: The edition documents not just what was found, but how it was processed — making the "vibe coding / promptotyping" workflow auditable
  • DH best practice: Aligns with FAIR principles (Findable, Accessible, Interoperable, Reusable) by exposing data lineage

Output Path Logic¶

Source paths are mirrored to docs/documents/, dropping the done/ workflow directory:

sources/QGW/Vienna_1177-1414_ready/done/100.xml
  → docs/documents/QGW/Vienna_1177-1414_ready/100.html

Document Count Note¶

The plan mentions 3,530 source files but only 3,041 have the done/ marker. The remaining ~489 files are not in done/ directories and are excluded from the edition build (no done/ directory in their path).

GitHub Pages Deployment¶

  1. Run python -m edition build
  2. Commit docs/ to the repository
  3. In GitHub repo settings → Pages → Source: main branch, /docs folder
  4. The site is served at https://<user>.github.io/<repo>/
