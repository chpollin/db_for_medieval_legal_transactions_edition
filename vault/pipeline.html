Pipeline and Data Model (XSLT, Python, ER Model)¶

Summary¶

The authoritative transformation pipeline is the Python/lxml implementation (pipeline/, 14 transformer modules + 2 validation modules, 313 regression tests, 101,948 data rows). The legacy XSLT layer (analysis_3/, 21 XSLTs, Saxon HE 10.5) is retained as an archived regression reference only. The ER model (79 nodes, 103 edges) is documented below.

Pipeline Architecture¶

Phase 1 -- Inventory and master data: 1. filenames_to_csv.xsl -> filenames.csv (critical dependency for all subsequent steps) 2. personlist_to_csv.xsl -> persons.csv 3. orglist_to_csv.xsl -> organisations.csv 4. placelist_to_csv.xsl -> places.csv

Phase 2 -- Events: 5. events_to_csv.xsl -> events_in_sources.csv

Phase 3 -- Entities in events: 6-8. persons/orgs/places_in_events_to_csv.xsl

Phase 4 -- Entities in sources: 9-11. persons/orgs/places_in_sources_to_csv.xsl

Phase 5 -- Relations (multiplexed): 12. person_relations_in_sources_to_csv.xsl x 6 types (kin, occ, title_ref, rep, friend, buis) 13. topo_rel_to_csv.xsl x 3 types (topo, tenant, owner) 14. transactiongoods_in_events_to_csv.xsl

Empirical output statistics (row counts, diff analysis_3 vs. data_export): see Mapping XSLT -> Python table below.

Transformation Logic per XSLT¶

A: Master Data Extraction (directly from registers)¶

filenames_to_csv.xsl: - Input: personList.xml (as entry point for collection()) - XPath: collection(concat($sourceuri, '?select=*.xml;recurse=yes')) -- scans all XMLs in sources/ - CSV columns: id; file; collection; subcollection; authority; editor; status; projekt; date - Includes: process-date.xsl

personlist_to_csv.xsl: - Input: personList.xml (directly, no filenames.csv needed) - XPath: tei:listPerson//tei:person - CSV columns: id; sex; forename_reg; forename_orig; surname_reg; surname_reg_source; surname_orig; addname_orig; addname_reg; genname; authority; note; dead_before; PAGEID_WienWiki; Name_WienWiki; editor - Extracts: @xml:id, @sex, tei:persName/tei:forename/tei:reg|tei:orig, tei:persName/tei:surname/tei:reg|tei:orig, tei:persName/tei:addName, tei:persName/tei:genName, tei:idno[@type='URI'], tei:note, tei:death/@notAfter, tei:idno[@type='PAGEID_WienWiki'], @resp

orglist_to_csv.xsl: - Input: orgList.xml (directly) - XPath: tei:listOrg//tei:org - CSV columns: id; name_reg; name_reg_source; name_orig; type; observance; authority; sub; org_key; place_key; editor - Extracts: @xml:id, tei:orgName/tei:reg|tei:orig, @type, tei:trait/tei:desc, tei:idno, hierarchy (ancestor::tei:org), tei:location/tei:address/tei:addrLine/@corresp, @resp

placelist_to_csv.xsl: - Input: placeList.xml (directly) - XPath: tei:listPlace//tei:place - CSV columns: id; name_reg; name_reg_source; name_orig; type; lat; lng; rel_place_key; Adresse_ak; geonames; parzelle_idno; authority; sub; place_key; editor - Extracts: @xml:id, tei:placeName/tei:reg|tei:orig, @type, tei:location/tei:geo (lat/lng via substring-before/after), tei:location//tei:addrLine, tei:idno[contains(.,'geonames')], tei:idno[@type='parzelle'], hierarchy (ancestor::tei:place), @resp

B: Entities in Events¶

Common pattern: Read filenames.csv via unparsed-text(), filter on status = 'done', load each source file via doc(), extract tei:rs[@type='event'][not(parent::tei:p[@rend='Nachtrag'])], then nested entities.

persons_in_events_to_csv.xsl: - XPath: $file//tei:body//tei:rs[@type='event'] -> .//*//tei:rs[@type='person'] - CSV columns: id; event_key; person_key; event_role; cert_to_person_key; event_catchword; file_key

orgs_in_events_to_csv.xsl: - XPath: $file//tei:body//tei:rs[@type='event'] -> ./tei:rs/tei:rs[@type='org'] - CSV columns: id; event_key; org_key; event_role; cert; event_catchword; file_key

places_in_events_to_csv.xsl: - XPath: $file//tei:body//tei:rs[@type='event'] -> ./tei:rs/tei:rs[@type='place'] - CSV columns: id; event_key; place_key; event_role; event_catchword; file_key

C: Entities in Sources¶

persons_in_sources_to_csv.xsl: - XPath: $file//tei:body//tei:*[@type='person' or contains(@corresp, 'pe__')] - CSV columns: id; person_key; kind_of_linking; source_text; cert_to_person_key; source_titles; cert_to_source_titles; source_prof; cert_to_source_prof; source_dead; file_key; doubletten_helper - Notable feature: Detects persons both via @ref and via @corresp (dual strategy)

orgs_in_sources_to_csv.xsl: - XPath: $file//tei:body//tei:*[@type='org' or contains(@corresp, 'org__')] - CSV columns: id; org_key; kind_of_linking; source_text; cert_to_org_key; file_key; doubletten_helper

places_in_sources_to_csv.xsl: - XPath: $file//tei:body//tei:*[@type='place' or contains(@corresp, 'pl__')] - CSV columns: id; place_key; kind_of_linking; source_text; file_key; doubletten_helper

D: Event Extraction¶

events_to_csv.xsl: - XPath: $file//tei:rs[@type='event'] - CSV columns: id; catchwords; event_key; date_not_before; date_not_after; event_in; date_orig; text; file_key - External dependencies: filenames.csv + normalisation_lists/label_norm_matching.csv (catchword normalisation) - Includes: process-date.xsl

E: Relation Extraction (parameterised)¶

person_relations_in_sources_to_csv.xsl: - Parameter: $relationtype (kin, occ, title_ref, rep, friend, buis) - XPath: $file//tei:body//tei:roleName[@type=$relationtype] - CSV columns: id; person_key; {relationtype}; cert_to_related_key; related_key; select_key; event_key; file_key - Extracts: ancestor::tei:rs[@type='person' or @type='org'] for context, @corresp for target entity

topo_rel_to_csv.xsl: - Parameter: $relationtype (topo, tenant, owner) - XPath (topo/tenant): $file//tei:roleName[@type=$relationtype] - XPath (owner, special mode): $file//tei:rs[@type='event'] -> Recipient/Issuer -> Transaction-Good-Places - CSV columns: id; place_key; cert_place_key; {relationtype}; rel_key; cert_topo_relation_key; event_key; xml_key - External dependencies (owner mode): filenames.csv + events_in_sources.csv + normalisation_lists/label_norm_matching.csv - Includes: process-date.xsl

transactiongoods_in_events_to_csv.xsl: - XPath: $file//tei:body//tei:rs[@type='event'] -> ./tei:rs[contains(@role, 'transactiongood')][not(tei:rs)] - CSV columns: id; event_key; transactiongood; source_catchword; file_key

F: Not in Standard Pipeline¶

widma_topo_rels_to_csv.xsl: - Project-specific (WIDMA). Extracts place relations with filter on tei:add[./text() = 'next_to_widmer']. - Not called in transform_all.sh.

org_regs_from_csv_to_xml.xsl: - Reverse transformation: CSV -> XML. Updates tei:orgName elements from CSV data. - Not called in transform_all.sh.

persons_events_to_csv_special.xsl: - Alternative variant of persons_in_events. Has its own shell script (person_in_events_to_csv_special.sh), but not included in transform_all.sh.

G: Utility¶

process-date.xsl (232 lines): - Shared template library, included via xsl:include in filenames, events, topo_rel. - Named template createdate: Normalises TEI date attributes (@notBefore, @notAfter, @from, @to, @when) to ISO format.

Common Processing Pattern¶

All source-text-based XSLTs (everything except master data) follow this pattern:

  1. Load filenames.csv via unparsed-text()
  2. Tokenise each line, filter on status = 'done'
  3. Load source file via doc(concat($source_path, collection, '/', subcollection, '/done/', filename))
  4. Extract TEI elements with specific XPath
  5. Output CSV row with semicolon delimiter

Register input: The shell scripts pass personList.xml, orgList.xml, or placeList.xml as input. This serves as the entry point for resolve-uri() -- the actual source texts are loaded via doc(). Which register is passed is functionally irrelevant, as long as the relative path to sources/ and analysis_3/output/ resolves correctly.

Technical Patterns¶

  • XSLT 2.0 throughout (version="2.0")
  • Text output with semicolon as CSV delimiter ($delim = ';')
  • unparsed-text() to read CSV (filenames.csv, label_norm_matching.csv, events_in_sources.csv)
  • collection() for recursive scanning of source texts (only in filenames_to_csv.xsl)
  • Parameterisation: $relationtype for multiplexed transformations
  • Duplicate detection: doubletten_helper columns in persons/orgs/places_in_sources
  • Certainty tracking: @cert attributes are consistently propagated
  • Addendum filter: [not(parent::tei:p[@rend='Nachtrag'])] excludes addenda from event extraction

Potential Problems¶

  1. Several shell scripts use personList.xml as input for non-person transformations (serves as entry point for resolve-uri(), not as data input).
  2. No error handling in shell scripts.
  3. Hardcoded paths to indices/ and sources/.
  4. Saxon JAR excluded via .gitignore -- pipeline not runnable without manual JAR provision.
  5. 3 commented-out steps in transform_all.sh: earlier versions via orgList/placeList were replaced by versions via personList, reason not documented.
  6. topo_rel_to_csv.xsl in owner mode has complex logic with 3 external CSV dependencies -- error-prone and difficult to maintain.
  7. analysis_3/src/shell/README.md contains copy-paste errors: persons_in_events documented as a duplicate of events, typo "Bescheibung" (2x), encoding artefact. The README is unreliable as a documentation source -- the actual logic must be read from the shell scripts themselves.

Debugging Insights (XSLT vs. Python)¶

  • normalize_space(): XSLT only handles XML whitespace (#x20,#x09,#x0A,#x0D), Python \s also matches U+2028
  • lxml load_dtd=False: Does not normalise attribute whitespace like Saxon
  • XPath .//*//: Requires 2+ nesting levels (not .//)
  • text()[not(parent::tei:add)]: Only text with an immediate add parent element is excluded
  • Saxon base-uri(): URL-encodes spaces as %20

Debugging Insights (lxml / RelaxNG)¶

  • xmlns:xsi prefix must be declared on the <grammar> element, otherwise etree.RelaxNG() fails with "no namespace for prefix xsi"
  • etree.RelaxNG() only accepts XML syntax, not Compact Syntax
  • etree._RelaxNG is not a public class — use hasattr(obj, "validate") instead of isinstance()
  • RelaxNG validation of 3,530 files completes in <1 second
  • RelaxNG forbids duplicate attribute definitions on a single element — attribute groups that conflict with per-element attributes must be excluded (see project.md Section 2)

Colour coding: Core entities (blue, #3366FF): event, person, organisation, place. Attributes (orange, #FF6600): title, profession, date, etc. Functional roles (green): other, witness, issuer/donor, recipient. Relationship types (magenta): staff, kinship, title, occupation.

Each node contains in its GraphML description attribute the TEI and CEI encoding rules side by side: - Person: <rs type="person" ref="pe__*"/> (TEI) / <cei:persName id="p*..."/> (CEI) - Organisation: <rs type="org" ref="org__*"/> (TEI) / <cei:orgName id="o*..."/> (CEI) - Event: <rs type="event" ref="ev__*"/> (TEI)

The CEI notation (Charter Encoding Initiative) is a relic from the Monasterium/MOM origin of the regesta. It is no longer used in the current workflow but remains documented in the data model.

Edges as directed arrows without cardinalities -- not a standard ER diagram but an informal relationship diagram. Organised in 4 groups: list of persons, list of organisations, list of places, list of events.

Empirical Cardinalities¶

The following cardinalities were determined empirically from the data (not shown in the GraphML diagrams):

Relation Cardinality Explanation
Source -> Event 1:n One source contains multiple events
Event -> Person (fn) n:m Multiple persons in multiple roles per event
Event -> Org n:m Multiple organisations per event
Person -> roleName 1:n Multiple roles per person per source
Person -> Person (kin) n:m Kinship relations
Person -> Org (occ) n:m Occupational relations
Person -> Org (title_ref) n:m Title references
